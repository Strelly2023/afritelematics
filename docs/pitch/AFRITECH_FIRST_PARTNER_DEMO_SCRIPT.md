# AfriTech First Partner Demo Script

Status: FIRST PARTNER DEMO SCRIPT
Classification: LIVE_PARTNER_DEMO_EXECUTION_SURFACE

Purpose: define the first live partner demo using the architecture compliance
dashboard, replay-backed operator view, Trust Explorer, and AfriCPPT protocol
surfaces.

This demo script is a bounded commercial and technical presentation surface.
It is not a production proof packet by itself.

## Demo Goal

Show a partner that AfriTech can:

- execute a real operational flow
- prove what happened through replay
- show system adherence to architecture
- expose bounded verification through AfriCPPT
- package the result into a partner-safe explanation surface

## Demo Narrative

```text
real action
-> replay-backed proof
-> architecture compliance
-> partner verification
-> adoption pathway
```

## Demo Sequence

### 1. Opening Position

State clearly:

- AfriTech is not selling a dashboard only
- AfriTech is selling replay-backed trust infrastructure
- the UI explains
- the protocol exchanges evidence
- replay proves

### 2. Show The Architecture Compliance Dashboard

Use:

- Architecture Compliance Dashboard
- System adherence to architecture
- Architecture Test Status
- Drift Detection Report

Explain:

- the system shows adherence to architecture live
- drift is checked for new modules not in architecture
- drift is checked for orphan components
- drift is checked for undocumented flows

### 3. Show Replay And Evidence Control

Use:

- Replay Health
- Evidence Health
- Guard Violations
- Trust Metrics

Explain:

- trust state comes from replay, evidence, and guard signals
- UI is a projection surface only
- replay remains the admissible truth authority

### 4. Show Trust Explorer

Use:

- Trust Explorer Registry
- Verification Visualization
- First 5 Partners
- Protocol Components

Explain:

- public visibility does not become truth authority
- registry indexes evidence only
- verification views resolve to replay-linked packets

### 5. Show AfriCPPT Integration Surface

Use:

- `docs/standards/AFRICPPT_PROTOCOL_SPEC.md`
- `POST /v1/partner/verify`
- `GET /v1/partner/anchors/{anchor_id}`
- `POST /v1/trust/network/verify`

Explain:

- partners integrate with bounded evidence
- external verification APIs expose verification outcomes only
- SDK / integration adapters simplify access but do not redefine truth

### 6. Close On Pilot Motion

Offer the next step:

- bounded pilot
- named operational corridor
- architecture compliance export
- partner onboarding kit
- first integration sprint

## Demo Success Criteria

- partner understands the authority boundary
- partner sees architecture compliance as a live discipline
- partner can name its verification workflow integration point
- partner accepts a bounded pilot as the next step

## Non-Claims

This script does not permit this claim:

```text
the demo UI itself certifies truth
public registry visibility replaces replay
partner demo completion proves production readiness
```
