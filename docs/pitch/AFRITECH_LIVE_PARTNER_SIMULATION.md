# AfriTech Live Partner Simulation

Status: LIVE PARTNER SIMULATION
Classification: COMMERCIAL_STRESS_REHEARSAL_SURFACE

Purpose: rehearse a realistic partner conversation where the buyer pushes hard
on need, trust, cost, risk, and adoption while AfriTech responds using live
runtime proof surfaces.

This simulation is a rehearsal surface.
It is not proof that a real partner has already converted.

## Simulation Goal

Prepare the AfriTech team to hold position under pressure without drifting into:

- inflated readiness claims
- abstract platform language
- unverifiable promises

## Hard-Push Simulation

### Partner Push 1

```text
We already have logs. Why do we need this?
```

Response:

```text
We are not replacing logs with prettier logs. We are showing replay, signed
receipt, external verification, and trust SLA on one bounded workflow. If that
workflow is already solved with confidence and low cost, you should not buy
this. If it is not solved, this is how you verify it live.
```

### Partner Push 2

```text
Why should we trust your verification endpoint?
```

Response:

```text
Do not trust the endpoint blindly. Use the verification packet, inspect the
authority boundary, and compare it to the signed receipt and replay surfaces.
The endpoint is a bounded proof exchange surface, not a request for blind trust.
```

### Partner Push 3

```text
Your pricing is high for a pilot.
```

Response:

```text
The pilot is not access pricing. It buys one verified workflow, one signed
proof path, one external verification path, and one agreed success metric. If
that workflow is not worth the pilot fee, we should not force the engagement.
```

### Partner Push 4

```text
What happens if your trust SLA goes to BREACH?
```

Response:

```text
That is why the SLA surface exists. BREACH is visible, not hidden. The runtime
still gives replay, evidence, guard alerts, and signed receipts for diagnosis.
We are selling explainable trust degradation, not silent opacity.
```

### Partner Push 5

```text
We do not want a platform replacement project.
```

Response:

```text
Neither do we. The first step is one bounded workflow, one owner, one pilot,
and one runtime verification path. This is additive until the workflow value is
proven.
```

## Simulation Rule

Always answer with:

- one workflow
- one endpoint
- one proof artifact
- one bounded next step
