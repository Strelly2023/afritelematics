# AfriRide Trust Explorer Go-Live Checklist

Status: TRUST EXPLORER GO-LIVE CHECKLIST
Classification: PUBLIC_LAUNCH_EXECUTION_CHECKLIST

Purpose: define the operational checklist required before the Trust Explorer is
launched publicly.

This checklist is a launch control surface.
It is not proof that the public launch has already happened.

## Go-Live Rule

```text
the Trust Explorer may go live only when public evidence panels are accurate,
bounded, and explicitly non-authoritative
```

## Product Checklist

- Trust Explorer Registry panel visible
- Verification Visualization panel visible
- First 5 Partners panel visible
- Monetization Surface panel visible
- Protocol Components panel visible
- authority boundary copy visible
- no mutation controls exposed

## Evidence Checklist

- curated registry entries loaded
- sample verified packet checked
- sample rejected packet checked
- replay-linked wording verified
- registry indexes evidence wording verified
- not a second truth layer wording verified

## Technical Checklist

- API base URL configured for public demo mode
- dashboard renders on desktop
- dashboard renders on mobile
- read-only contract preserved
- instrumentation headers remain bounded for demo use
- no hidden operator override flows exposed

## Launch Operations Checklist

- launch owner assigned
- rollback owner assigned
- public demo script prepared
- support contact path prepared
- first protocol inquiry response path prepared

## Go / No-Go Questions

1. Can a viewer see a registry entry without mistaking it for runtime truth?
2. Can a viewer understand verification visualization without internal database access?
3. Can the team explain one verified packet and one rejected packet live?
4. Can the team state the authority boundary in one sentence?

## Authority Boundary

This checklist permits only this bounded claim:

```text
AfriRide has an execution-ready go-live checklist for the public Trust Explorer
launch
```
